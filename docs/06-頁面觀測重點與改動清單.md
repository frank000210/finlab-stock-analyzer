# 頁面觀測重點與改動清單

> 搭配 `04-開發指引.md` 使用。本文件定義「每個頁面該回答什麼問題」，以及既有頁面目前離這個目標還差什麼。

---

## 一、原則

1. **每個頁面只對應一個觀測重點**（一句話講清楚：這頁是要讓使用者看懂什麼）。首頁例外，不受此規範限制。
2. **先寫觀測重點，再設計版面**：新增頁面時，先寫好這句話，再決定要放哪些圖表/卡片；不要先堆圖表再回頭找理由。
3. **每個頁面頂部必須用 `PageFocusBanner` 元件顯示這句話**（`frontend/src/components/PageFocusBanner.vue`），放在頁面根 `<div>` 的第一個子元素：
   ```html
   <div class="xxx-page">
     <PageFocusBanner text="這頁的觀測重點一句話" />
     ...
   </div>
   ```
4. **加新圖表前先自問**：這個圖表是在回答本頁的觀測重點，還是另一個頁面的問題？如果是後者，應該放到對應頁面，而不是塞進目前這頁。
5. **改動時同步更新本文件**：頁面用途改變（觀測重點改了）或新增/移除頁面時，回來更新下面的總表，避免文件與程式碼脫節。

## 三、現況比對與待改清單

以下是重新檢視每個頁面「現有內容」對照「觀測重點」後，發現尚未完全對齊、需要補強或調整的項目。沒有列出的頁面代表現況內容已經符合觀測重點，暫不需要改動。

### OverviewView（觀測重點：綜合健康度總覽）
- 現況：雷達圖 + 各維度（季節性/領先落後/主力/社群/基本面）摘要卡片，符合「總覽」定位。
- **待新增**：每日漲跌熱力圖（Treemap），放在雷達圖之前作為最上層總覽，資料來源為當日大盤/自選股 OHLC 與市值。這是目前唯一缺的「市場當下狀態」視角——現有卡片都是「這檔股票的各面向分數」，還沒有「整體市場今天長怎樣」的一眼概觀。

### GraphView（觀測重點：關聯強度與網絡結構）
- 現況：力導向網絡圖，看得出「跟誰連動」的結構，但看不出精確的相關係數數值。
- **待新增**：相關係數熱力圖（矩陣式），作為網絡圖的另一個檢視模式（可用 tab 切換），資料為同一組標的的歷史報酬率矩陣。

### AnalysisView（觀測重點：技術面是否支持進出場時機）
- 現況：K線+均線、RSI/MACD/KD、布林通道、ADX/ATR/OBV 已完整覆蓋動能與波動面向。
- **待新增**：成交量分佈圖（Volume Profile，價位 × 量的橫向柱狀圖），疊加在 K 線圖右側或下方，用來標出主力真正的成本區與套牢區。這是目前技術面缺的「價位結構」視角——現有指標都是時間序列，沒有「哪個價位量最大」的視角。

### PublicDataView（觀測重點：判斷股價相對便宜或昂貴）
- **現況與觀測重點落差最大的頁面**：目前內容是重大公告清單、歷年配息表、最新財務摘要，本質是「資料陳列」，不是「便宜或昂貴的判斷」。使用者看完這頁還是答不出「現在貴不貴」。
- **待新增（優先）**：本益比／股價淨值比河流圖（PE/PB Band），把股價疊在歷史估值區間帶上，資料為 EPS/淨值 + 歷史股價。
- **待新增**：月營收年增率（YoY）柱狀圖疊股價走勢，資料為月營收公告，用來看基本面轉折是否領先股價反應。
- 既有的公告/配息/財務摘要可以保留，但建議降到頁面下半部，作為輔助明細，把河流圖與營收圖放在最上方，才符合「先回答觀測重點」的頁面設計原則。

### 其餘頁面（Decision / Rotation / Backtest / Seasonal / Lead-Lag / Major-Players / Chip / Social-Buzz / Settings / TradeDashboard / AI-Signals / Risk-Monitor / Data-Agent / Trade-Approval / Signal-Rules / Admin）
- 現況內容與各自觀測重點一致，暫無需調整的項目。

## 四、優先順序建議

1. PublicDataView 補上 PE/PB 河流圖 + 營收成長圖（現況落差最大，且是常見的基本面判斷工具）。
2. OverviewView 加每日漲跌熱力圖（使用頻率最高的頁面，補上市場總覽視角效益最大）。
3. AnalysisView 加成交量分佈圖。
4. GraphView 加相關係數熱力圖（優先度較低，網絡圖已能達成大部分需求）。
